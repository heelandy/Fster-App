PART 1 — CORE PRODUCT (FOSTER-PARENT APP)
==================================================================
Goal: a private app for foster parents to manage placement information,
appointments, documents, routines, expenses, contacts and licensing —
simple, secure, mobile-friendly, for real daily use.

------------------------------------------------------------------
CORE FEATURES
------------------------------------------------------------------
1. Child Profile Management ✅
   Name / preferred name, age, date of birth, placement date, case number,
   caseworker, school, doctor, allergies, medications, important notes,
   emergency contacts, placement status. Sensitive fields are access-controlled.

2. Appointments & Calendar ✅
   Doctor, therapy, dental, court dates, school meetings, caseworker visits,
   home inspections, licensing deadlines. Reminders + calendar views.

3. Document Storage ✅
   Secure upload/storage of placement paperwork, medical/school/court/
   licensing documents, receipts, case notes. Private + access-controlled.

4. Daily Care Logs ✅
   Meals, sleep, behavior, mood, school updates, visits, medical concerns,
   incidents, milestones. Date-based history.

5. Medication Tracking ✅
   Name, dosage, schedule, start/end date, prescribing doctor, notes,
   missed doses. Includes reminders and a give/miss/refuse log.

6. Expense Tracking ✅
   Clothing, school supplies, food, hygiene, activities, transportation,
   medical, childcare, room setup, receipts. Monthly summaries + CSV export.

7. Contact Management ✅
   Caseworker, licensing worker, GAL, attorney, therapist, doctor, dentist,
   teacher, school counselor, biological family, emergency contacts —
   with phone, email, agency, notes and role.

8. Routines & Checklists ✅
   Reusable morning / after-school / bedtime / visit-day / school-day /
   intake / home-inspection / medication checklists. Mark tasks complete.

9. Licensing & Compliance Tracker ✅
   Training hours, fingerprinting, background checks, home-study items,
   fire inspection, medical forms, pet vaccinations, CPR/First Aid,
   annual renewal deadlines, required documents. Shows upcoming deadlines.

PART 2 — ADMIN PLATFORM (25-section spec)
==================================================================
The admin console lives at /admin in its own route group, gated by an ADMIN
JWT + a granular permission matrix (lib/admin.ts). 11 tabs: Overview · Users ·
Tickets · Analytics · Finance · Notifications · Settings · Integrations ·
System · Security log · Admin log.

 1. Admin purpose (control center) ✅
    One dashboard to manage the whole app without direct database access.

 2. Admin user roles (7 tiers) ✅
    SUPER_ADMIN · ADMIN · MANAGER · SUPPORT · MODERATOR · FINANCE_ADMIN ·
    READ_ONLY. (Consolidated capability/restriction table is in PART 3.)

 3. Admin login system ✅ 🟡
    ✅ email+password, 2FA (TOTP) with scannable QR, password reset,
       login-attempt limits, IP logging, session timeout, forced logout,
       admin activity logging, AND a per-device active-session list with
       selective revoke (/account → Active sessions).
    🟡 no separate admin-only login page (admins use the same hardened
       /login; protection is 2FA + RBAC, not a distinct URL).

 4. Admin dashboard ✅
    Tiles: total/active/new users, suspended, revenue, failed payments,
    open tickets, security alerts (24h), server/database/API status,
    recent admin actions. ("Understand the app in <30s.")

 5. User management module ✅
    Search/filter; view account metadata; create user (emails set-password
    link); edit name/email; reset password (admin triggers the link, never
    sees the password); verify account; suspend; ban; reactivate; unlock;
    add internal note; assign role; soft-delete (deactivate + tombstone,
    restorable). All actions audited old→new. Never exposes hashes/child data.

 6. Admin management (manage admins) ✅
    Create admin (create-user + adminRole, needs admins.manage) · set/change
    admin role · disable (suspend) · force password reset · limit permissions
    (role matrix) · view admin activity (admin audit log) · remove admin access.

 7. Permission system (granular) ✅
    adminCan() matrix enforced per route: users.view/edit/suspend/delete/note,
    admins.manage, payments.view/refund, settings.update, content.moderate,
    support.manage, analytics.view, system.view, logs.view, reports.export.

 8. Content management ➖
    Not applicable — a private foster-care app has no public/user-generated
    content (posts, products, lessons) to moderate.

 9. Reports & moderation ➖
    Not applicable — no public UGC or user-on-user reporting. (User safety is
    handled via Support tickets, §11, and account actions, §5.)

10. Payments & subscription module ✅
    Finance tab: recent payments, subscriptions by tier/status, revenue,
    payment issues. In-app Stripe refunds (full/partial) and account credits.
    Mark payment issue (via status) · export financial report (CSV).
    🟡 plan catalogue is code+Stripe-config (no plan-CRUD UI); coupons via Stripe.

11. Support ticket module ✅
    User threads (/support) + staff Tickets tab: reply, set status
    (OPEN/PENDING/RESOLVED/CLOSED), priority. Admin notified on new ticket.

12. Settings module ✅
    System settings: app name, maintenance mode, signups enabled, email-
    verification required. Integrations tab (SUPER_ADMIN + TOTP step-up):
    live Stripe keys/prices + email config, no code/env edits.

13. Analytics & reporting ✅
    DAU/WAU/MAU, stickiness, signups & active-user charts (inline SVG), churn,
    revenue, KPIs. CSV report export (users/subscriptions/revenue).

14. Audit log system ✅
    AdminAuditLog + SecurityAuditLog: actor, action, target, old/new value,
    IP, timestamp. Append-only — immutable (no edit/delete route exists).

15. Notification system ✅
    Internal admin alerts on new signup, failed payment, new support ticket;
    unread counter in the console.

16. Security architecture ✅ 🟡
    HTTPS/HSTS, 2FA, strong passwords, session expiry, role permissions, IP
    logging, rate limiting, CSRF, XSS (nonce CSP), SQLi-safe (Prisma),
    AES-256-GCM encryption at rest, admin action logs, no direct DB editing.
    🟡 backup system is operational (see §22), not application code.

17. Database structure ✅
    users/admin-roles, support_tickets, settings, notifications, invites,
    reset/verification tokens, payments, subscriptions, audit_logs — all present.

18. Backend architecture (services) ✅
    Clean lib/ layer: auth, authz, admin, totp, tokens, email, stripe,
    billing-sync, notify, settings, audit, rate-limit, storage, config, etc.

19. Frontend admin panel ✅
    11 pages/tabs (see header). One web interface controls the app.

20. API structure (admin routes) ✅
    Every /api/admin/* route is auth + permission gated: users (GET/POST),
    users/[id] (PATCH/DELETE), payments, payments/[id]/refund, payments/credit,
    reports/export, tickets, analytics, health, settings, notifications, audit,
    step-up, integrations.

21. System health module ✅ 🟡
    DB latency, storage usage, memory, uptime, integration status, config
    warnings. 🟡 CPU/uptime are process-level (no external infra metrics).

22. Backup & recovery 🟡
    Operational, documented in DEPLOYMENT.md (managed Postgres backups /
    pg_dump cron, encrypted storage, restore testing) — not application code.

23. Admin workflow ✅
    Login (2FA) → check dashboard alerts → review new users → answer tickets →
    check failed payments → review system/security → export reports → review
    audit logs → log out. Supported by the tabs above.

24. Recommended tech stack ✅
    Documented variant chosen: Next.js route handlers (not NestJS), NextAuth
    (not a separate JWT service), Resend email, local private storage (S3/R2
    pluggable), optional Redis rate limit, optional error-reporting webhook.

25. Final architecture summary ✅
    ~22/25 areas fully implemented; §8/§9 not applicable; §22 operational.

PART 3 — ADMIN CAPABILITIES, RESTRICTIONS & ROLE TIERS
==================================================================
(Consolidates the former "What an Admin CAN/SHOULD NOT Do" and the three

------------------------------------------------------------------
WHAT AN ADMIN CAN DO  — all implemented
------------------------------------------------------------------
User management ✅ — view accounts · create users · edit info · suspend ·
  ban · reset passwords (trigger link) · unlock · verify accounts · assign roles.
Support operations ✅ — view tickets · respond · escalate (priority) · close.
Business operations ✅ — view subscriptions/invoices/payments · issue refunds ·
  apply credits · 🟡 manage plans (code/Stripe config).
Security operations ✅ — view login history (security log) · view suspicious
  activity · force-logout sessions · lock compromised accounts (suspend/ban).
Reporting & analytics ✅ — view analytics · export reports (CSV) · monitor
  KPIs · track revenue.
System monitoring ✅ — view system health/server status/logs · monitor
  performance · monitor errors (optional error-webhook).
Content management ➖ — N/A (no UGC; see §8/§9).

------------------------------------------------------------------
WHAT AN ADMIN MUST NOT DO  — all enforced
------------------------------------------------------------------
✅ Never see passwords/PINs/secret questions/auth tokens — only bcrypt hashes
   exist; admin can only trigger a password reset.
✅ Never read private child/medical/legal data — requireHousehold has no admin
   bypass; admin views are aggregate counts only; data encrypted at rest.
✅ Audit logs immutable — append-only; no edit/delete/hide path.
✅ Cannot bypass security controls — no toggle disables logging/monitoring.
✅ No direct production-DB access — everything goes through guarded routes.
✅ Payment-provider secrets restricted — Stripe keys encrypted, masked,
   SUPER_ADMIN + TOTP step-up only; encryption keys are env-only.
✅ Financial info masked / no raw card data — card entry is Stripe-hosted.
✅ No permanent hard-delete of records — soft delete / deactivate / restore.

------------------------------------------------------------------
ADMIN ROLE TIERS ✅  (7 implemented; capabilities via the permission matrix)
------------------------------------------------------------------
SUPER_ADMIN  — everything: create admins, change permissions, configure system,
               integrations. Cannot see passwords, delete audit logs, or
               decrypt protected data without authorization.
ADMIN        — manage users, content moderation, daily operations; not critical
               system settings unless granted.
MANAGER      — view reports, supervise; limited access to sensitive data.
SUPPORT      — view users, edit basic info, reset passwords, handle tickets;
               no finance / security / infrastructure.
MODERATOR    — handle reports & user safety actions; no payments / settings / DB.
FINANCE_ADMIN— view payments, process refunds, generate financial reports;
               cannot modify users or infrastructure.
READ_ONLY    — view dashboards/reports only; no changes.
(➖ "System/Infrastructure Admin" is an operational ops role, not an in-app tier.)

------------------------------------------------------------------
AUDIT-LOG REQUIREMENT ✅  (single source of truth)
------------------------------------------------------------------
Every important action is recorded — login, account changes, role/permission
changes, subscription/payment changes (refund, credit), settings changes,
suspension/deletion. Each entry stores actor, action, target, old value, new
value, date/time, IP (and device where available). Nothing is editable after
it is logged.

⬜ AI features — placement-matching recommendations, missing-document alerts,
   compliance monitoring, risk scoring, report generation, onboarding assistant.
⬜ Reimbursement/stipend financials & tax forms for foster parents.

✅ GOLDEN RULE (privacy principle) — already enforced today:
   A foster parent sees only what's needed to care for their child; no one,
   including administrators, can view passwords, delete audit history, or
   bypass legal/privacy protections. Foster-care data is highly sensitive
   child-welfare information and is protected accordingly.

PART 5 — SECURITY THREAT MODEL & DEFENSE STATUS
==================================================================
(Sorted from the appended attack taxonomy. Each threat is marked with the
app's current defense: ✅ defended in app code · 🟡 partial / needs an option
enabled · ⬜ operational / infrastructure (not app code) · ➖ not applicable.)
Verified against: middleware CSP, next.config security headers, lib/authz,
lib/auth, lib/request (CSRF), lib/storage, lib/crypto, Stripe webhook handling.

------------------------------------------------------------------
PRIORITY DEFENSES FOR A FOSTER-CARE APP (highest first)
------------------------------------------------------------------
 1. Broken access control ........ ✅ household-scoped RBAC, no admin bypass
 2. IDOR ......................... ✅ every query joins householdId; 404 cross-tenant
 3. SQL injection ................ ✅ Prisma parameterized; no raw SQL
 4. XSS .......................... ✅ React escaping + nonce CSP + email escapeHtml
 5. CSRF ......................... ✅ Origin/Referer check (no fail-open) on mutations
 6. Account takeover ............. ✅ bcrypt(12), lockout, rate limit, TOTP 2FA, forced logout
 7. API authorization ............ ✅ per-route capability + admin permission matrix
 8. File upload .................. ✅ MIME+magic-byte, size cap, private store, authed download
                                   🟡 no malware/AV scan
 9. Insider threats .............. ✅ admins can't see child data/passwords; immutable audit
10. Cloud misconfiguration ....... ⬜ deploy-time (private buckets/DB, least-priv IAM)
11. Data leaks ................... ✅ generic errors, private storage, secrets in env
12. Ransomware ................... ⬜ operational (backups + server hardening)
13. DDoS ........................ ⬜ operational (CDN/WAF, e.g. Cloudflare)
14. Supply chain ................ 🟡 minimal pinned deps; needs audit + Dependabot
15. Social engineering .......... 🟡 2FA blunts phished passwords; rest is process

------------------------------------------------------------------
FULL THREAT TAXONOMY → DEFENSE
------------------------------------------------------------------
1.  Authentication attacks ✅
    Brute force / spraying / credential stuffing → account lockout + login
    rate limit (in-memory, optional Redis) + failed-login tracking. Session
    hijack/fixation → signed httpOnly JWT, session timeout, tokenVersion
    forced-logout. JWT forgery → signed w/ NEXTAUTH_SECRET; role/tokenVersion
    re-read from DB each request. MFA → TOTP 2FA (QR + backup codes).
    Reset/magic-link abuse → hashed single-use 1h tokens, anti-enumeration.
    (OAuth ➖ — credentials provider only.)

2.  Authorization attacks ✅
    Privilege escalation (H/V), IDOR/BOLA, broken access control, forced
    browsing, API permission bypass → deny-by-default capability matrix,
    household-scoped queries with no admin bypass, requireUser reads role
    fresh, admin permission matrix enforced per route.

3.  Injection attacks ✅
    SQLi → Prisma parameterized. Command/OS → no shell exec from input.
    Email-header/CRLF → email sent via Resend JSON API (no raw SMTP headers).
    NoSQL/LDAP/XPath/XML ➖ — not used (PostgreSQL only).

4.  Cross-site scripting (XSS) ✅
    Stored/reflected/DOM → React auto-escaping, strict nonce CSP (script-src
    'self' 'nonce' 'strict-dynamic'), object-src 'none', base-uri 'self';
    email HTML escaped (escapeHtml).

5.  CSRF ✅
    assertSameOrigin (trusted Origin, else trusted Referer, else 403 — no
    fail-open) on every mutation; SameSite cookies; CSP form-action 'self'.

6.  API attacks ✅ 🟡
    BOLA/function-level → scoped + permission matrix. Mass assignment → Zod
    whitelists fields. Excessive data exposure → explicit select (no hashes /
    child data). 🟡 rate-limit is per-instance unless Upstash Redis is enabled.
    GraphQL ➖ (REST only).

7.  Business-logic attacks ✅ 🟡
    Payment/subscription bypass → entitlements derive ONLY from signed Stripe
    webhooks (idempotent/dedup). Refunds/credits → admin-only + audited.
    Coupons/trials → enforced by Stripe. 🟡 plan-limit checks can race under
    concurrency (low impact).

8.  File-upload attacks ✅ 🟡
    Web shell/executable/polyglot/type-bypass → MIME + magic-byte validation,
    size pre-check, files stored OUTSIDE the web root and served only via an
    authenticated, ownership-checked route with attachment + nosniff.
    🟡 no malware/AV scanning (add ClamAV / a scanning API for untrusted files).

9.  Denial of service (DoS) 🟡 ⬜
    App-layer → rate limiting + bounded body/field sizes. ⬜ true DoS
    resilience (slowloris, floods) belongs to a CDN/WAF + infra.

10. Distributed DoS (DDoS) ⬜
    Operational — front with a CDN/WAF (Cloudflare/Fastly); not app code.

11. Session attacks ✅
    Hijack/fixation/prediction → signed httpOnly SameSite JWT, rotation on
    privilege change, tokenVersion revocation, session timeout, AND a
    per-device session list with selective revoke (UserSession rows; a revoked
    device fails requireUser on its next request). "Sign out everywhere" also
    revokes all rows. (/account → Active sessions.)

12. Browser-based attacks ✅
    Clickjacking → X-Frame-Options: DENY + CSP frame-ancestors 'none'. MIME
    sniffing → X-Content-Type-Options: nosniff. Referrer leakage →
    Referrer-Policy. Powered-by hidden; Permissions-Policy locks camera/mic/geo.

13. DNS attacks ⬜
    Operational — registrar lock, DNSSEC, monitor for sub-domain takeover.

14. Email-based attacks 🟡 ⬜
    App → escaped HTML, rate-limited reset/verification sends, anti-enumeration.
    ⬜ SPF/DKIM/DMARC are DNS records to set for the sending domain.

15. Supply-chain attacks 🟡
    Minimal, version-pinned dependencies (lockfile); the QR encoder added is a
    single zero-dependency package. 🟡 enable Dependabot / `npm audit` in CI and
    review the existing transitive advisories (Next.js tree).

16. Cloud attacks ⬜
    Deploy-time — private object storage (no public buckets), non-public DB,
    least-privilege IAM, no container privilege. App keeps S3 keys private +
    validates storage keys.

17. Database attacks ✅ 🟡
    Enumeration/extraction → parameterized access, least-privilege app role,
    AES-256-GCM field encryption for sensitive child/medical fields + 2FA secret.
    🟡 backup encryption + DB-credential rotation are operational.

18. Cryptographic attacks ✅ 🟡
    AES-256-GCM, bcrypt(12), CSPRNG (crypto.randomBytes), HMAC-signed tokens,
    HSTS (prod). 🟡 ENCRYPTION_KEY must be backed up + rotated off-app; TLS/cert
    management is operational.

19. Malware attacks ⬜
    Operational — server hardening, EDR/AV, offline backups for recovery.
    (Upload AV scanning is the app-side piece — see §8.)

20. Insider threats ✅ 🟡
    Admins cannot read child data or passwords; least privilege; every admin
    action audited to an immutable log; no direct DB editing. 🟡 SUPER_ADMIN is
    powerful (mitigated: no child-data/password/decryption access; TOTP step-up
    for secrets).

21. Social engineering 🟡
    2FA limits damage from phished passwords; support tickets warn against
    sharing PII. Rest is people/process (verification scripts, staff training).

22. Mobile-app attacks ➖
    No native mobile app today (responsive web only). Revisit if one ships.

23. Infrastructure attacks ⬜
    Operational — patching, firewall, least exposed services, config hardening.

24. Bot attacks 🟡
    Rate limiting + signup toggle + email verification cut fake/abusive signups.
    🟡 no CAPTCHA / bot-scoring on register & login (add if abuse appears).

25. Data-exposure attacks ✅ 🟡
    Generic error responses (no stack traces), private storage, no debug output
    in prod, secrets in env, audit metadata carries no child PII. 🟡 backup/log
    storage privacy is operational.
